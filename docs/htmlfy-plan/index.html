Context

The user generates HTML deliverables across many repos (audits, reports, explainers, FOR_DUMMIES guides, slide decks, call-chain maps). Commit 7ce1e8a in the dotclaude submodule formalized this with a single line in ~/.claude/CLAUDE.md:

"Prefer self-contained HTML over markdown for any document deliverable…"

…and migrated 6 user-owned skills to HTML output. Two gaps remained:

  1. No standardized workflow for ad-hoc HTML asks that don't fit an existing specialized skill (for-dummies, game-design, etc.). Each one re-invented inline CSS conventions, output-location choices, and structural decisions.
  2. No sharing path. All artifacts landed on local disk. No central place to send a link from. This is the actual pain point.

The html-effectiveness reference adds 4 net-new ideas beyond the current CLAUDE.md line: (a) spatial reasoning beats prose, (b) side-by-side comparison layouts, (c) interactivity tightens iteration loops, (d) multi-file linked collections.

Locked-in answers from planning:
  • Sensitivity: mixed — per-artifact public/private routing required.
  • GH plan: verify before deciding the central-repo architecture.
  • Shape: skill + publish command, decoupled.

Shared substrate verdict

None for the general-HTML case. Each skill defines its own inline CSS. Only sui-marp-theme has a shared asset (assets/sui-theme.css), and it's Sui-branding-specific. But critically, for-dummies (the most mature) already contained a complete "HTML Output Conventions" block at for-dummies/SKILL.md:340-354 covering self-containment, semantic tags, inline SVG, system fonts, max-width, no JS. That block was the de-facto template and was duplicated implicitly across the others — now lifted into references/html-conventions.md.

Artifact lifecycle

  • 76 committed .html files in the repo, mostly plugin test fixtures. Only 4 user-generated outputs in .claude/ subdirs.
  • No .gitignore rule targets *.html. Skills currently write to the target project's root or sub-package directory.
  • Zero publishing infrastructure: no .github/workflows/, no gh-pages branches, no docs/ folders with Jekyll configs anywhere.

External-reference delta

html-effectiveness — three-bucket breakdown
  • Already in CLAUDE.md: "prefer self-contained HTML for document deliverables" (the baseline).
  • Already embodied in a skill: the "HTML Output Conventions" block at for-dummies/SKILL.md:340-354, which literally linked to thariqs.github.io/html-effectiveness at line 354.
  • Net-new (now encoded in html-conventions.md):
    1. Spatial > prose — when relationships are spatial, render boxes + arrows.
    2. Side-by-side comparison — when comparing options, render them adjacent, not sequentially.
    3. Interactive tightening of iteration loops — collapsibles, copy buttons, anything that lets readers explore at their own pace.
    4. Multi-file linked collections vs monolithic single-file outputs.
GitHub Pages quickstart — operational facts
  • Branch + /docs folder is a viable publishing source on main. Personal sites at username.github.io; project sites at username.github.io/repo/.
  • Quickstart is silent on private-repo Pages — the critical fact for the "mixed sensitivity" case. Historically that requires GitHub Pro/Team.
  • 10-minute publish delay; _config.yml (Jekyll) needed only for theming.
  • No granular access control: a Pages site is either fully public or — on a paid plan with private repo — visible only to repo collaborators.

Recommendation

Build both a generic skill and a separate publish skill, plus a shared conventions reference. Decoupled by design — never auto-publish.

1. New skill html-artifact

A small, focused skill in dotclaude:.claude/skills/html-artifact/ for the ad-hoc case. Owns four things:

  1. The medium-fit decision. Apply html-effectiveness's 4 criteria. If markdown wins, say so.
  2. The shared conventions reference. Cross-links to references/html-conventions.md; no inline CSS rules duplication.
  3. Default output location. Next to the source by default; AskUserQuestion when ambiguous.
  4. A trigger description that defers to specialized skills per the superpowers:writing-skills doctrine.

2. Shared references/html-conventions.md

Lifted from for-dummies/SKILL.md:340-354 verbatim plus the 4 net-new html-effectiveness rules. Eliminates the implicit duplication across all 9 HTML-producing skills.

3. New skill publish-html

A separate, explicitly-invoked skill. Iron Law: never publish without a mandatory AskUserQuestion sensitivity gate. Routes:

SensitivityDestinationVisibility
Public-safealilloig/artifacts via GitHub PagesPublic web
SensitiveSecret gist via gh gist create --secretURL-only, unindexed
Abort—Local only

4. CLAUDE.md edit

One line added under the existing HTML preference, pointing to both skills. Keeps the global preamble lightweight.

5. Central artifacts repo (one-time, manual)

alilloig/artifacts — public repo, Pages enabled on main / /docs.

Validation steps

  1. Description triggers correctly. Three "should trigger" / three "should NOT trigger" prompts. for-dummies MUST route to for-dummies, not html-artifact.
  2. HTML renders offline. Open from /tmp/ with no network — must render fully.
  3. Publish flow happy path. Public-safe → URL returns 200 within 15 minutes.
  4. Privacy guard fires. Feed publish-html a confidential-named file → sensitivity AskUserQuestion MUST fire before any push.
  5. First-run plan check is one-shot. Second invocation should not re-call gh api user.
  6. Submodule pin consistency. git submodule status shows no + prefix after the pin bump.
  7. For-dummies retrofit behavior-preserving. Cross-reference must resolve and yield the full convention set (verified).

Open items to revisit during implementation

  • Whether publish-html should also mint an opaque slug (e.g. UUID) for public URLs to make confidential-feeling artifacts harder to guess. Default: human-readable slug from filename.
  • Whether html-artifact should adopt for-dummies's two-zone (tutorial / appendix) structural pattern as a default. Default: agnostic.
